# Architecture & Code Quality Documentation

## 🏛️ Overall Architecture

### Layered Architecture Pattern

```
┌─────────────────────────────────────────┐
│        PRESENTATION LAYER               │
│    (React Components, Pages)            │
├─────────────────────────────────────────┤
│        STATE MANAGEMENT LAYER           │
│    (TanStack Query, React Hooks)        │
├─────────────────────────────────────────┤
│        SERVICE LAYER                    │
│    (API Service, Axios)                 │
├─────────────────────────────────────────┤
│        DATA LAYER                       │
│    (TypeScript Types, Interfaces)       │
└─────────────────────────────────────────┘
```

### Benefits of This Architecture

1. **Separation of Concerns**: Each layer has a single responsibility
2. **Testability**: Easy to mock services and test components
3. **Reusability**: Services can be used across multiple components
4. **Maintainability**: Changes in one layer don't affect others
5. **Scalability**: Easy to add new features without complexity

## 🔄 Data Flow Pattern

### Complete Request-Response Cycle

```
User Action (Click)
         ↓
Component Event Handler
         ↓
Hook Mutation/Query Call
         ↓
TanStack Query
         ↓
API Service (axios)
         ↓
Request Interceptor (add JWT token)
         ↓
Backend API
         ↓
Response Interceptor
         ↓
Handle Response/Error
         ↓
Cache Update
         ↓
Component Rerender
         ↓
UI Updated
```

### Example: Creating a Client

```typescript
// 1. User submits form
<button onClick={() => form.submit()}>Save</button>

// 2. Form handler
onSubmit(data) → createMutation.mutate(data)

// 3. Hook mutation
useCreateClient() → useMutation({ mutationFn: ... })

// 4. APIService
apiService.createClient(data) → axios.post('/clients', data)

// 5. Response handling
onSuccess() → queryClient.invalidateQueries()

// 6. Cache update
queryClient → auto-refetch useClients()

// 7. UI update
clients data changes → component rerenders
```

## 🎯 Key Design Patterns Used

### 1. Container/Presentational Components

**Container Components** (Pages):
- Manage state and logic
- Fetch data via hooks
- Pass data to presentational components

```typescript
// DashboardPage.tsx - CONTAINER
export function DashboardPage() {
  const { data: stats } = useDashboardStats();
  
  return (
    <MainLayout>
      <StatsCards stats={stats} />
      <RecentTasksTable tasks={stats.recentTasks} />
    </MainLayout>
  );
}
```

**Presentational Components** (UI):
- Pure functions, receive data as props
- No side effects or API calls
- Reusable across different containers

```typescript
// StatsCard.tsx - PRESENTATIONAL
interface StatsCardProps {
  title: string;
  value: number;
  icon: React.ReactNode;
  bgColor: string;
}

export function StatsCard({ title, value, icon, bgColor }: StatsCardProps) {
  return <Card>...</Card>;
}
```

### 2. Custom Hooks Pattern

**Why Custom Hooks?**
- Encapsulate logic
- Reusable across components
- Easier testing

```typescript
// Using individual hooks
const { data: clients } = useClients();
const createMutation = useCreateClient();
const deleteMutation = useDeleteClient(id);

// All hooks return { data, isLoading, error, isPending, mutate }
```

### 3. Query Key Factory Pattern

**Prevents Query Key Collisions**

```typescript
const queryKeys = {
  all: ['query'],
  clients: () => [...queryKeys.all, 'clients'],
  clientsList: (filters) => [...queryKeys.clients(), 'list', filters],
  client: (id) => [...queryKeys.clients(), id],
}

// Usage
useQuery({
  queryKey: queryKeys.client(id),
  queryFn: () => getClient(id),
})

// Invalidate all clients
queryClient.invalidateQueries({ queryKey: queryKeys.clients() })

// Invalidate specific client
queryClient.invalidateQueries({ queryKey: queryKeys.client(id) })
```

### 4. Error Boundary Pattern

**Interceptor-Based Error Handling**

```typescript
// API Service handles 401
if (response.status === 401) {
  → Try refresh token
  → If fail → Redirect to login
}

// Components display error states
if (error) {
  return <ErrorMessage error={error} />
}
```

### 5. Optimistic Updates Pattern

**Better UX:**

```typescript
// Option 1: Invalidate and refetch (used here)
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: [...] })
}

// Option 2: Optimistic update (for real-time apps)
onMutate: async (newData) => {
  queryClient.setQueryData(key, newData);
}
```

## 🔐 Security Implementation

### Authentication Flow

```
1. Login
   • User submits email + password
   • Form validation (Zod)
   ↓
2. Backend validation
   • Email lookup
   • Password verification
   ↓
3. JWT Generation
   • Access token (short-lived)
   • Refresh token (long-lived)
   ↓
4. Token Storage
   • Access: In memory (lost on refresh)
   • Refresh: HttpOnly cookie (secure)
   ↓
5. Subsequent Requests
   • Request interceptor adds token
   • Token in Authorization header
   ↓
6. Token Expiry
   • 401 response from backend
   • Response interceptor triggers refresh
   • New token returned
   • Original request retried
```

### Why HttpOnly Cookies?

- **XSS Protection**: JavaScript can't access token
- **CSRF Protection**: Same-site policy prevents cross-origin attacks
- **Automatic**: Browser sends cookie with each request

### Validation Strategy

```
Frontend (UX):           Backend (Security):
Zod validation           Express validators
Type checking            Server-side validation
Real-time feedback       Sanitization (mongo-sanitize)
Prevent invalid submit   Reject bad requests
```

## 📋 Code Quality Checklist

### Before Committing

- [ ] TypeScript: No `any` types
- [ ] Component: Pure function or custom hook
- [ ] Props: All required props documented
- [ ] Error: Error state handled
- [ ] Loading: Loading state shown
- [ ] Accessibility: Semantic HTML, ARIA labels
- [ ] Responsiveness: Mobile, tablet, desktop
- [ ] Performance: No unnecessary re-renders

### Component Template

```typescript

import { useQuery } from '@tanstack/react-query';
import { Card } from '@/components/ui/card';

  if (error) return <div>Error: {error.message}</div>;

  return (
    <Card>
      {/* Component JSX */}
    </Card>
  );
}
```

## 🎯 Best Practices Summary

### Do's ✅

- Use TypeScript strict mode
- Create reusable components
- Separate logic from rendering
- Handle loading/error states
- Validate user input
- Use meaningful variable names
- Document complex logic
- Keep components focused
- Use custom hooks for logic

### Don'ts ❌

- Don't use `any` types
- Don't make huge components
- Don't fetch in render
- Don't ignore console errors
- Don't hardcode strings
- Don't create prop drilling chains
- Don't use index as key
- Don't fetch without error handling

## 📈 Scaling the Application

### Adding New Features

1. **Create feature folder**
   ```
   features/newFeature/
   ├── NewFeatureForm.tsx
   ├── NewFeatureDialog.tsx
   └── hooks.ts (if needed)
   ```

2. **Add API methods**
   ```typescript
   // services/api.ts
   async getNewFeatures() { ... }
   async createNewFeature(data) { ... }
   ```

3. **Add query hooks**
   ```typescript
   // hooks/useQueries.ts
   export function useNewFeatures() { ... }
   export function useCreateNewFeature() { ... }
   ```

4. **Create page**
   ```typescript
   // pages/NewFeaturePage.tsx
   export function NewFeaturePage() { ... }
   ```

5. **Add route**
   ```typescript
   // App.tsx
   <Route path="/newfeature" element={<ProtectedRoute><NewFeaturePage /></ProtectedRoute>} />
   ```

### Migration Path

- Small project (current) → Medium project:
  - Add tests
  - Implement error boundaries
  - Add error logging service

- Medium project → Large project:
  - Add state management (Zustand if needed)
  - Implement advanced caching
  - Add analytics
  - Implement feature flags
